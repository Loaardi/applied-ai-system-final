# PawPal+ — AI-Powered Pet Care Assistant

## Original Project

PawPal+ evolved from the **PawPal Scheduler** built during Modules 1–3 of AI110. The original project was a Python-based pet care task manager that let owners create pets, assign care tasks (feeding, walking, grooming), and view sorted daily schedules. It featured recurring task support, conflict detection, and a Streamlit web interface — but all scheduling logic was manual and rule-based.

## Summary

PawPal+ transforms the original task scheduler into an **AI-powered pet care assistant**. Instead of manually adding every task, owners can now describe their pet and receive intelligent, personalized care recommendations generated by an AI agent. The system uses **Retrieval-Augmented Generation (RAG)** to look up breed-specific care data from a local knowledge base before generating advice, and an **agentic workflow** where the AI plans, retrieves, generates, and validates its own output. This matters because busy pet owners shouldn't need to research every care detail themselves — the assistant does it for them, grounded in real pet care knowledge rather than generic guesses.

## Architecture Overview

![System Architecture](assets/system_architecture.png)

The system follows a five-layer pipeline:

1. **Streamlit UI** — The user enters owner/pet information, manages tasks manually, or asks the AI assistant for help. This is the single entry point for all interactions.

2. **AI Agent** (`ai_agent.py`) — The agentic core. When a user requests a care plan, the agent: (a) parses the request, (b) retrieves relevant knowledge from the RAG knowledge base, (c) reasons about the pet's needs and schedule constraints, and (d) generates a personalized daily plan with explanations. Responses are grounded in the knowledge base data rather than an external language model.

3. **Knowledge Base / RAG** (`knowledge_base.py`) — A local JSON data store containing breed-specific care information: feeding schedules, exercise requirements, grooming frequency, health notes, and enrichment ideas for common pet types. The agent queries this store by species and breed before generating any recommendation, ensuring responses are grounded in structured data rather than hallucinated.

4. **Scheduler** (`pawpal_system.py`) — The original core logic layer. Tasks generated by the AI or added manually flow through the Scheduler, which handles time-based sorting, recurring task expansion, conflict detection, and filtering by pet or status.

5. **Validation Layer** — Three levels of quality assurance run in parallel:
   - **Automated tests**: 38+ pytest cases covering scheduling logic, recurrence, conflicts, and edge cases.
   - **Guardrails and logging**: Runtime error handling, input validation, and structured logging throughout the AI pipeline.
   - **Human review**: The user always reviews AI-generated suggestions before they become active tasks. The AI recommends; the human decides.

Data flows top-down (input → agent → scheduler → display) with a feedback loop: users can edit or reject AI suggestions, and those adjustments inform future interactions within the session.

## Setup Instructions

### Prerequisites

- Python 3.9 or higher
- An Anthropic API key (get one at [console.anthropic.com](https://console.anthropic.com))

### Installation

```bash
# 1. Clone the repository
git clone https://github.com/Loaardi/applied-ai-system-final.git
cd applied-ai-system-final

# 2. Create and activate a virtual environment
python -m venv .venv
# On Windows:
.venv\Scripts\activate
# On macOS/Linux:
source .venv/bin/activate

# 3. Install dependencies
pip install -r requirements.txt

# 4. Set your API key as an environment variable
# On Windows (PowerShell):
$env:ANTHROPIC_API_KEY="your-api-key-here"
# On macOS/Linux:
export ANTHROPIC_API_KEY="your-api-key-here"

# 5. Run the Streamlit app
streamlit run app.py
```

### Running Tests

```bash
python -m pytest tests/test_pawpal.py -v
```

## Design Decisions

**Why RAG instead of just prompting the AI?** Without a knowledge base, the AI would rely entirely on its training data for pet care advice, which can be generic or occasionally wrong. By retrieving structured, vetted care data for specific breeds before generating a response, the system produces more accurate and specific recommendations. The knowledge base also makes the system auditable — you can inspect exactly what data informed each recommendation.

**Why an agentic workflow?** A simple "send prompt, get response" approach wouldn't integrate with the existing scheduler. The agent pattern lets the AI plan its approach (retrieve knowledge → check existing schedule → generate recommendations → validate for conflicts), which produces coherent multi-step outputs instead of one-shot answers that ignore context.

**Why keep the original Scheduler?** The rule-based Scheduler is deterministic and well-tested. Rather than replacing it with AI-driven scheduling (which would be unpredictable), the AI generates *suggestions* that flow through the proven Scheduler for conflict detection and time sorting. This hybrid approach gets the best of both: creative AI recommendations with reliable scheduling guarantees.

**Why human-in-the-loop?** AI suggestions are never auto-committed. The user always reviews and approves. This is a deliberate guardrail — pet care has real consequences, and an AI shouldn't unilaterally change a medication schedule or remove a vet appointment.

**Trade-offs:**
- The local JSON knowledge base is limited to pre-loaded breeds. A production system would use a vector database or API. This keeps the project simple and reproducible without external dependencies beyond the API key.
- Using the Anthropic API means the app requires an internet connection and an API key. An offline alternative (like a local model) would remove this dependency but significantly reduce output quality.
- The agent currently operates within a single session — it doesn't remember past interactions. Adding persistence would improve the experience but adds complexity beyond the project scope.

## Testing Summary

### What Worked

- **Scheduler logic is solid**: 38+ pytest cases pass consistently, covering recurring tasks, conflict detection, time-window queries, sorting edge cases, and filter combinations.
- **RAG retrieval is deterministic**: Given a species and breed, the knowledge base always returns the same structured data, making AI inputs predictable and testable.
- **Conflict detection catches real problems**: The system reliably flags overlapping tasks and the AI uses this information to suggest fixes.

### What Didn't

- **AI output varies between runs**: Even with the same input, the AI generates slightly different wording each time. This makes exact-match testing impossible — the test suite checks for key content (task types, time ranges, breed-specific terms) rather than exact strings.
- **Edge cases in multi-pet scheduling**: When three or more pets have overlapping care needs, the AI occasionally suggests schedules that are technically conflict-free but unrealistic for a single owner to execute (e.g., three tasks within a 15-minute window).
- **Knowledge base coverage gaps**: If a user enters an uncommon breed or exotic pet not in the knowledge base, the system falls back to generic species-level advice, which is less useful.

### What I Learned

- Testing AI systems requires a different mindset than testing deterministic code. You test for *properties* (does the response mention the right breed? does it include exercise recommendations?) rather than exact outputs.
- RAG dramatically improves response quality compared to raw prompting. The difference between "Dogs need walks" and "Golden Retrievers need 1-2 hours of daily exercise and are prone to hip dysplasia" is the knowledge base.
- Guardrails matter more than features. Logging, error handling, and input validation caught more bugs during development than any single feature addition.

## Reflection

Building PawPal+ taught me that the hardest part of AI engineering isn't calling an API — it's designing the system around the AI so it behaves reliably. The original PawPal was a straightforward scheduling app where every output was predictable. Adding AI introduced uncertainty: the same question might produce different answers, the model might ignore retrieved data, or it might suggest something that sounds good but conflicts with existing tasks.

This forced me to think about AI as a component in a larger system, not as the system itself. The Scheduler, knowledge base, guardrails, and human review layer all exist to constrain and validate the AI's output. The AI is powerful but not trustworthy on its own — it needs structure around it.

The RAG implementation was the most valuable learning experience. Seeing the concrete difference between grounded and ungrounded AI responses made the concept real in a way that reading about it never did. When the AI has access to breed-specific data, its advice is specific, actionable, and verifiable. Without it, the advice is generic and sometimes wrong. That gap is why RAG matters.

If I were to continue this project, I would add a vector database for the knowledge base (to handle any breed without pre-loading), implement session persistence so the AI remembers past interactions, and build a more robust evaluation framework that scores AI outputs against a rubric automatically. I would also explore fine-tuning a smaller model on pet care data to reduce API costs while maintaining quality.

This project changed how I think about building with AI: it's not about making the AI do everything — it's about making the AI do the right thing, in the right place, with the right checks around it.
